# 搜书机器人项目 - 完全重构计划

## 背景与现状

当前项目存在大量技术债务，多次修复未能解决根本问题。主要问题包括：
- 前端代码质量差（大量any类型、重复代码、不一致的HTTP客户端）
- 后端路由与控制器不匹配（多个路由指向不存在的方法）
- 安全漏洞（密钥暴露、CORS完全开放、弱密码）
- Nginx配置问题导致API请求无法正确路由

## 重构目标

构建一个可维护、可扩展、安全的搜书机器人系统。

## 技术栈

### 后端
- **框架**: Laravel 11 (PHP 8.2+)
- **数据库**: SQLite (保持现状，便于迁移)
- **认证**: Laravel Sanctum + Token过期机制
- **对象存储**: Cloudflare R2 (保持)
- **API风格**: RESTful + JSON API规范
- **测试**: PHPUnit单元测试

### 前端
- **框架**: Vue 3 + TypeScript (严格模式)
- **构建工具**: Vite
- **状态管理**: Pinia
- **路由**: Vue Router 4
- **HTTP客户端**: 统一的useApi composable (基于fetch)
- **UI组件**: 纯CSS + 少量通用组件（移除Vuetify）
- **类型**: 完整的TypeScript类型定义

## 关键改进点

### 1. 后端改进
- **移除ScanController**: 不再扫描本地目录，所有书籍通过上传API存入R2
- **统一错误响应**: 所有API返回统一格式 `{success, data, message}`
- **Token过期**: Sanctum Token设置7天过期
- **API限速**: 登录/注册接口单独限速(5次/分钟)
- **CORS安全**: 只允许特定域名，不允许`*`
- **密钥管理**: 使用环境变量，不提交到版本控制

### 2. 前端改进
- **TypeScript严格模式**: 所有组件使用lang="ts"
- **统一HTTP客户端**: 所有请求通过useApi
- **类型安全**: API响应类型定义
- **组件复用**: 提取公共组件
- **路由懒加载**: 使用动态导入
- **错误处理**: 统一的错误提示机制

### 3. 部署改进
- **Docker Compose**: 一键启动所有服务
- **Nginx配置**: 正确的SPA路由和API代理
- **SSL证书**: 自动续期
- **日志管理**: 轮转和集中管理

## 重构计划评审与补充

> 基于对现有代码的全面审查（6个Controller、6个Model、R2StorageService、所有数据库迁移、前端核心代码），以下是评审发现和补充建议。

### 一、与现有代码不一致的问题（必须修正）

#### 1. API响应格式不统一
现有代码混用两种格式：
- AuthController / BookController / AdminController：`{err: 'ok', data: {...}, msg: '...'}`
- BookUploadController：`{message: '...', errors: {...}}`

重构计划提出统一为 `{success, data, message}`，但应明确：
- **建议**：统一为 `{code: 0, data: {}, message: ''}` 格式，code=0表示成功，非0为错误码
- 需要一个全局响应辅助函数或中间件来强制统一

#### 2. Laravel版本差异
现有项目是 **Laravel 10**，计划要求 **Laravel 11**。Laravel 11 有重大目录结构变化：
- 无 `Http/Kernel.php`，中间件改用 `bootstrap/app.php` 注册
- 路由文件改为 `routes/web.php` + `routes/api.php`（默认只有web.php）
- 配置文件精简

**建议**：创建全新 Laravel 11 项目，从零实现，不要尝试升级。

#### 3. books表字段遗漏
计划中的 books 表设计缺少现有代码中已有的字段：

| 字段 | 用途 | 建议 |
|------|------|------|
| `category` | 书籍分类 | **保留**，分类浏览需要 |
| `tags` | 标签 | **保留**或改为关联表 |
| `description` | 书籍描述 | **保留** |
| `cover_path` | 封面路径 | **保留**，改为 `cover_url` |
| `upload_id` | 分片上传ID | **保留**，分片上传核心字段 |
| `upload_parts` | 分片信息(JSON) | **保留**，分片上传核心字段 |
| `uploader_id` | 上传者ID | **保留**，用于追踪上传来源 |
| `indexed_at` | 索引时间 | **可移除**，不再扫描本地目录 |

#### 4. 路由遗漏
现有代码中有但计划中缺失的路由：

- `GET /api/user [auth]` - 获取当前用户（计划写的是 /api/auth/me，需统一）
- `GET /api/profile [auth]` - 获取用户详情
- `PUT /api/profile [auth]` - 更新用户资料
- `GET /api/downloads [auth]` - 获取下载记录
- `POST /api/books/batch-upload-dedup [auth]` - 带去重的批量上传
- `GET /api/books/file/{token}` - 预签名文件下载（无需Auth Header）
- `POST /api/admin/login` - 管理员独立登录（不走 /api/auth/login）

#### 5. 卡密生成格式不统一
- AuthController: 4段4位大写字母格式（如 `ABCD-EFGH-IJKL-MNOP`）
- AdminController: md5(uniqid()) 前16位（16位十六进制）

**建议**：统一为4段4位大写字母格式，更易识别和输入。

#### 6. 部署方式冲突
计划使用 Docker Compose，但服务器已有 1Panel 管理的 OpenResty/PHP-FPM/MySQL 容器。

**建议**：继续使用 1Panel 方式部署，理由：
- 服务器已稳定运行 1Panel 管理的服务
- Docker Compose 与 1Panel 容器会端口/网络冲突
- 1Panel 方式更简单，适合此项目规模
- 部署步骤：PHP-FPM容器运行Laravel + OpenResty代理 + 前端构建产物放入public目录

### 二、安全方面补充

#### 1. 密码策略加强
现有：最少6位。建议：最少8位，必须包含字母和数字。
验证规则改为：`required|string|min:8|regex:/^(?=.*[A-Za-z])(?=.*\d).+$/`

#### 2. Token过期实现方案
Sanctum 默认不过期，需要在 `config/sanctum.php` 中设置：
`'expiration' => 60 * 24 * 7`（7天，单位：分钟）
同时需要前端处理 401 响应自动跳转登录页。

#### 3. 管理员操作二次验证
敏感操作（删除用户/书籍、修改系统设置）应要求重新输入密码：
`POST /api/admin/verify-password {password}` - 验证通过后发放临时token（5分钟有效）

#### 4. 搜索防注入
SQLite LIKE 查询需转义特殊字符（%和_），scopeSearch 方法中应先转义再拼接。

#### 5. 下载限速
现有代码有每日下载上限（download_limit 系统设置），计划中 API 的下载接口必须保留此逻辑。

#### 6. 上传权限
建议：普通用户也可上传，但需要管理员审核后才能公开（upload_status 增加 pending_review 状态）。

### 三、数据库设计补充

#### 1. 缺少索引
- downloads 表：`user_id + created_at` 联合索引（每日下载计数查询需要）
- books 表：`search_count` 索引（热门排序需要）
- cards 表：`type` 索引和 `used` 索引

#### 2. 软删除
users 和 books 表应增加 `deleted_at` 字段，使用 Laravel 的 SoftDeletes trait。
避免误删数据无法恢复，管理后台删除操作应先软删除，7天后可永久删除。

#### 3. file_path 字段处理
现有 books 表有 file_path（本地路径），重构后全部走 R2 存储：
- 迁移时：本地文件上传到 R2，然后移除 file_path 字段
- 如果本地文件量很大，迁移脚本需要支持断点续传

### 四、前端架构补充

#### 1. Vuetify 替代方案
移除 Vuetify 后需要明确替代：

| Vuetify 组件 | 替代方案 |
|-------------|---------|
| v-data-table | 自定义表格组件（纯CSS + 排序/分页逻辑） |
| v-form / v-text-field | 自定义表单组件 + vuelidate 验证 |
| v-dialog | 自定义 Modal 组件 |
| v-snackbar | 自定义 Toast 组件 |
| v-pagination | 自定义分页组件 |
| v-card | 自定义 Card 组件 |

**更推荐**：使用 Naive UI（Tree-shakable、TypeScript 原生支持、体积小）或 Headless UI + Tailwind CSS。

#### 2. 路由懒加载
所有路由组件改为动态导入：`const Home = () => import('@/views/Home.vue')`

#### 3. 国际化 (i18n)
现有代码有 `plugins/i18n.ts`，**建议保留**，后续可扩展多语言支持。

#### 4. 错误边界
App.vue 根组件应添加 ErrorBoundary 包裹 RouterView，防止子组件错误导致整页白屏。

### 五、部署方案补充

#### 推荐：继续使用 1Panel 方式
- OpenResty容器：静态文件 + API代理
- PHP-FPM容器：Laravel应用
- SQLite文件：`/opt/1panel/.../data/books.db`

#### 数据库路径修正
现有 .env 中 `DB_DATABASE=/app/data/books.db` 是 Docker 容器内路径，1Panel 部署应改为实际路径：
`DB_DATABASE=/opt/1panel/apps/openresty/openresty/www/sites/book/index/backend/data/books.db`

#### PHP版本
服务器 PHP 容器是 8.4.6，Laravel 11 要求 8.2+，完全兼容。

### 六、数据迁移补充

#### 1. 迁移前统计
先统计现有数据量：用户数、书籍数、本地存储的书（有file_path无r2_key）、R2存储的书（有r2_key）。

#### 2. file_path 到 R2 迁移
遍历所有有 file_path 但无 r2_key 的书籍，将本地文件上传到 R2，更新数据库记录。
上传 key 格式：`novels/migrated/{文件名}`

#### 3. 迁移回滚方案
- 迁移前完整备份 SQLite 数据库
- 保留旧代码目录（重命名为 backend_old）
- 新旧系统使用不同数据库文件，验证通过后再切换
- 切换失败：改 Nginx 配置指回旧目录即可

### 七、功能补充

#### 1. 搜索优化
SQLite LIKE 对大数据量性能差，建议使用 FTS5 全文搜索：
- 创建虚拟表 `books_fts USING fts5(title, author, content=books, content_rowid=id)`
- 添加同步触发器，INSERT/UPDATE/DELETE 时自动同步

#### 2. 管理后台批量操作
- `DELETE /api/admin/users/batch {ids: [1,2,3]}`
- `DELETE /api/admin/books/batch {ids: [1,2,3]}`

#### 3. 操作审计日志
新增 audit_logs 表，记录用户操作（action、entity、entity_id、old_value、new_value、ip），便于追踪问题。
